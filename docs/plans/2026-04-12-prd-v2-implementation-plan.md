# PRD V2 实施计划

## 背景

旧版 `prd` 的内容骨架是好的，尤其是以下几点值得保留：

- `用户故事 -> 范围 -> 业务对象 -> 业务逻辑 -> 页面与菜单 -> 角色与权限` 这条主线很稳
- `业务逻辑` 拆成 `主流程 / 分支与异常 / 状态机 / 业务规则分类` 很适合交给 `trd`
- 复杂视图调用 `view-spec`、复杂流程调用 Mermaid 图示的思路是对的
- 用户故事、字段和规则保留来源追溯的做法有价值

问题主要不在内容结构，而在运行机制还停留在旧体系：

- 依赖 `hub` 传 page_id
- 没有明确绑定到现在的 FlowUS 结构
- 更像一次性写文档，而不是“先总后分、逐块共创”的过程
- 未决问题没有完全统一到业务域根页面下的 `待确认事项`

## 目标

重写一个全新的 `prd-v2`，与旧版并存，满足以下目标：

- 无 `hub`，直接依赖 FlowUS 恢复上下文
- 最终权威内容写入当前 EPIC 页面下的 `PRD`
- 所有未决问题统一进入当前业务域根页面下的页面 `待确认事项`
- 保留旧版 PRD 的高价值章节结构
- 主流程改成“先总后分、先讨论再写回”的共创模式
- 让 `trd` 和 `uat` 读完后基本不需要再猜业务逻辑

## 上游输入

`prd-v2` 的上游输入应固定为：

- `全局资源`
- 当前业务域根页面下的 `BRD`
- 当前业务域根页面下的 `Solution`
- 当前业务域根页面下的页面 `待确认事项`
- 当前 EPIC 页面
- 当前 EPIC 下已有的 `PRD`（如果存在）

缺失处理原则：

- 缺 `全局资源`：说明可信度受限，必要时暂停
- 缺 `BRD`：停止，先回 `discovery`
- 缺 `Solution`：停止，先回 `solution`
- 缺业务域根页面下的页面 `待确认事项`：先创建
- 缺 `PRD`：进入新建模式

## 运行模式

### 模式 1：新建模式

适用于：

- 当前 EPIC 下还没有 `PRD`
- `PRD` 只有标题或很薄的骨架
- 用户明确说“从头写这个 EPIC 的 PRD”

目标：

- 先建立 EPIC 总览和范围
- 再逐块收敛出第一版 PRD

### 模式 2：增量修订模式

适用于：

- 已有 `PRD` 需要补规则、补状态、补权限、补视图
- `Solution` 更新后，需要反映到 `PRD`
- 用户对已有章节提出明确修订意见

目标：

- 精准更新某几个章节
- 不重写未变化内容

## 核心设计决策

### 1. 保留旧版主骨架

正文结构继续围绕：

- EPIC 目标与范围
- 用户故事
- 业务对象与信息
- 业务逻辑
- 页面与菜单
- 角色与权限
- 外部接口 / 报表打印 / 配置项（按需）

### 2. 强化“先总后分”

PRD 不直接从局部规则开写，而是按下面节奏推进：

1. 先列 EPIC 能力总览或闭环清单
2. 再拆用户故事
3. 再确认 `做什么 / 不做什么`
4. 再进入核心章节逐块细化

### 3. 业务逻辑继续使用四层写法

这是旧版最值得保留的部分：

- 主流程
- 分支与异常处理
- 状态机
- 业务规则分类

### 4. 视图规格不在 PRD 内硬写死

`PRD` 负责决定：

- 需要哪些视图
- 哪些页面是主工作界面
- 哪些视图需要详细规格

复杂页面应按需调用 `view-spec`，并把结果作为 `PRD` 中的视图说明或关联产物。

### 5. 统一未决问题入口

所有尚未确认但会影响 PRD 结论的问题，统一进入：

- 当前业务域根页面下的页面 `待确认事项`

`PRD` 本身只保留：

- `关键假设`
- `已放弃 / 暂缓方案`
- `下一步重点确认`

### 6. 支持阶段性审查

当本次对 `PRD` 有实质写回，且准备交给 `trd` 或 `uat` 使用时，`prd-v2` 应建议或按需调用 `review` 做快速审查。
