# Skill System Guidelines Design

## 背景

现有 skill 体系已经把主要意图表达出来了，但仍有几个明显问题：

- 过度依赖中心化调度思路
- 太强调“把文档产出来”，不够强调“把方案讨论出来”
- 文档过程缺少分块讨论和逐步确认
- skill 的无状态边界不够清晰

团队希望把系统调整为更轻、更直接、更适合小团队高频共创的模式。

## 本次设计结论

本次确定以下方向：

1. 取消 `hub` 作为中心调度入口
2. skill 改为直接触发或按当前任务自然调用
3. FlowUS 文档作为唯一外部记忆
4. skill 自身保持无状态
5. 文档是共识快照，不是 skill 的唯一目标
6. 核心 skill 必须采用分步共创而不是整篇生成
7. skill 结束时只建议下一步，不强制路由
8. 后续补一个通用 `review` skill 做横切质量门

## 备选思路比较

### 方案 1：保留 Hub，继续强化状态机

优点：

- 统一调度入口清晰
- 适合严格流程化推进

缺点：

- 对你们当前团队来说过重
- 容易把精力放在“状态维护”上，而不是“讨论收敛”上
- skill 更容易写成流程节点，而不是真正可独立使用的能力单元

### 方案 2：去 Hub，采用文档驱动的无状态 skill

优点：

- 更轻量
- 更符合 FlowUS 作为外部记忆的设定
- 更符合你们当前“持续讨论、持续完善”的工作方式
- skill 边界更清晰，更适合逐个优化

缺点：

- 对 skill 写法要求更高
- 需要统一文档结构，否则下游 skill 恢复上下文会不稳定

这是本次采用的方案。

### 方案 3：完全自由流，不再定义 skill 写法规范

优点：

- 最灵活

缺点：

- 很快会失去一致性
- 每个 skill 的触发、边界、输出都会飘
- 难以沉淀出稳定可复用的团队工作方式

不采用。

## 关键设计原则

### 1. 文档驱动

上游文档就是下游 skill 的提示词工程。
每个 skill 先读文档，再讨论，再写回。

### 2. 分步共创

skill 必须把“讨论”做成主流程。
推荐采用“读上下文 -> 复述理解 -> 分块讨论 -> 写回确认内容 -> 建议下一步”的结构。

### 3. 轻状态

不做复杂轮次管理，但每份关键文档必须保留：

- 当前共识
- 未决问题
- 关键假设
- 已放弃方案

### 4. 角色分层

- 域级文档负责认知收敛
- EPIC 级文档负责交付落地
- Review 负责横切审查
