# UAT V2 实施计划

## 背景

旧版 `uat` 的内容骨架是好的，尤其以下部分值得保留：

- 强调“任何人打开 Odoo 照着走一遍就能验收”
- `冒烟测试 -> 完整测试 -> 回归测试指引 -> 验收签字` 的结构非常实用
- 每条用例追溯到 `PRD / TRD` 的做法很有价值
- 很重视前置条件、测试账号、测试数据、权限隔离和肉眼可判断的结果

问题主要不在验收内容，而在运行机制还停留在旧体系：

- 依赖 `hub` 传 page_id
- 没有明确绑定到现在的 FlowUS 结构
- 更像一次性生成测试文档，而不是基于最新 `PRD / TRD` 持续增量更新
- 未决问题没有完全统一到业务域根页面下的 `待确认事项`

## 目标

重写一个全新的 `uat-v2`，与旧版并存，满足以下目标：

- 无 `hub`，直接依赖 FlowUS 恢复上下文
- 最终权威内容写入当前 EPIC 页面下的 `UAT`
- 所有未决问题统一进入当前业务域根页面下的页面 `待确认事项`
- 保留旧版 `UAT` 的高价值结构
- 主流程改成“先恢复范围和追溯关系，再按冒烟 / 完整 / 回归分层收敛”的模式
- 让测试、PM、客户、开发都能直接用

## 上游输入

`uat-v2` 的上游输入应固定为：

- `全局资源`
- 当前业务域根页面下的页面 `待确认事项`
- 当前 EPIC 页面
- 当前 EPIC 下的 `PRD`
- 当前 EPIC 下的 `TRD`（如果存在）
- 当前 EPIC 下已有的 `UAT`（如果存在）

缺失处理原则：

- 缺 `全局资源`：说明可信度受限，必要时暂停
- 缺 `PRD`：停止，先回 `prd`
- 缺 `TRD`：允许继续，但明确边界 case 可信度会下降
- 缺业务域根页面下的页面 `待确认事项`：先创建
- 缺 `UAT`：进入新建模式

## 运行模式

### 模式 1：新建模式

适用于：

- 当前 EPIC 下还没有 `UAT`
- `UAT` 只有标题或很薄的骨架
- 用户明确说“从头写这个 EPIC 的验收文档”

目标：

- 建立第一版人工可执行 `UAT`
- 先把测试环境、冒烟主线和追溯关系搭起来

### 模式 2：增量修订模式

适用于：

- 已有 `UAT` 需要补场景、补边界、补权限或补回归指引
- `PRD` 或 `TRD` 更新后，需要同步到 `UAT`
- 用户对已有章节提出明确修订意见

目标：

- 精准更新受影响章节
- 不重写未变化内容

## 核心设计决策

### 1. 保留旧版主骨架

正文结构继续围绕：

- 测试环境准备
- 冒烟测试
- 完整测试
- 回归测试指引
- 验收签字

### 2. 先定范围与追溯，再写测试步骤

`UAT` 不直接从 case 开始写，而是按下面节奏推进：

1. 先确认本次验收范围和范围排除
2. 再从 `PRD / TRD` 提取追溯矩阵
3. 先写冒烟主线
4. 再补完整测试分类
5. 再补回归和签字

### 3. 非研发可执行是第一原则

所有用例都要满足：

- 步骤写全
- 入口写清
- 预期结果能被肉眼判断
- 角色和账号清楚

### 4. 保留对 `TRD` 的补强作用

`PRD` 是主要输入，`TRD` 用来补：

- 边界值
- record rule 权限点
- 自动化和数据文件相关可测项
- 视图实现带来的可见性或交互细节

### 5. 支持 subagent 调用

`uat-v2` 可以被其他 skill 作为 subagent 调用。

被调用时：

- 不需要和用户来回对话
- 直接从 FlowUS 读 `PRD / TRD / 待确认事项`
- 发现不明确的地方优先继续完成产出，再把问题写入页面 `待确认事项`

### 6. 统一未决问题入口

所有尚未确认但会影响 `UAT` 结论的问题，统一进入：

- 当前业务域根页面下的页面 `待确认事项`

`UAT` 本身只保留：

- `关键假设`
- `已放弃 / 暂缓方案`
- `下一步重点确认`

### 7. 支持阶段性审查

当本次对 `UAT` 有实质写回，且准备交给业务或测试执行时，`uat-v2` 应建议或按需调用 `review` 做快速审查。
