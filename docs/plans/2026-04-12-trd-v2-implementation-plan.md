# TRD V2 实施计划

## 背景

旧版 `trd` 的技术内容骨架是好的，尤其以下部分值得保留：

- `模块规划 -> 模型定义 -> 业务逻辑 -> 权限 -> 视图实现 -> 数据文件 -> 测试指引` 的章节顺序很稳
- `PRD 规则 -> 技术实现映射表` 很有价值，能防止实现漏规则
- 技术层类图、模型来源对照表、联动逻辑标注这些内容很适合 `TRD`
- 权限、数据文件、测试指引都写到了可直接落代码的粒度

问题主要不在技术内容，而在运行机制还停留在旧体系：

- 依赖 `hub` 传 page_id
- 没有明确绑定到现在的 FlowUS 结构
- 更像一次性生成技术文档，而不是“先定模块与边界，再逐块共创”的过程
- 未决问题没有完全统一到业务域根页面下的 `待确认事项`

## 目标

重写一个全新的 `trd-v2`，与旧版并存，满足以下目标：

- 无 `hub`，直接依赖 FlowUS 恢复上下文
- 最终权威内容写入当前 EPIC 页面下的 `TRD`
- 所有未决问题统一进入当前业务域根页面下的页面 `待确认事项`
- 保留旧版 `TRD` 的高价值技术骨架
- 主流程改成“先模块规划和技术边界，再逐章落细节”的共创模式
- 让研发或 AI 读完后基本可以直接开工

## 上游输入

`trd-v2` 的上游输入应固定为：

- `全局资源`
- 当前业务域根页面下的 `Solution`
- 当前业务域根页面下的页面 `待确认事项`
- 当前 EPIC 页面
- 当前 EPIC 下的 `PRD`
- 当前 EPIC 下已有的 `TRD`（如果存在）

缺失处理原则：

- 缺 `全局资源`：说明可信度受限，必要时暂停
- 缺 `Solution`：停止，先回 `solution`
- 缺 `PRD`：停止，先回 `prd`
- 缺业务域根页面下的页面 `待确认事项`：先创建
- 缺 `TRD`：进入新建模式

## 运行模式

### 模式 1：新建模式

适用于：

- 当前 EPIC 下还没有 `TRD`
- `TRD` 只有标题或很薄的骨架
- 用户明确说“从头做这个 EPIC 的技术设计”

目标：

- 建立第一版可开发的 `TRD`
- 先把模块拆分、模型和实现边界讲清楚

### 模式 2：增量修订模式

适用于：

- 已有 `TRD` 需要补模型、补方法、补权限、补视图实现
- `PRD` 更新后，需要同步到 `TRD`
- 用户对已有章节提出明确修订意见

目标：

- 精准更新受影响章节
- 不重写未变化内容

## 核心设计决策

### 1. 保留旧版主骨架

正文结构继续围绕：

- 模块规划
- 模型定义
- 业务逻辑实现
- 权限
- 视图实现
- 数据文件
- 测试指引
- 外部接口（按需）

### 2. 先定技术骨架，再写实现细节

`TRD` 不直接从方法开始写，而是按下面节奏推进：

1. 先确认范围和模块拆分
2. 再确认模型和来源
3. 再把 `PRD` 规则翻成技术实现
4. 再补权限、视图、数据文件和测试

### 3. 业务逻辑继续使用映射表驱动

这是旧版最值得保留的部分：

- 先输出 `PRD -> TRD` 映射表
- 再按校验、计算、工作流、自动化、编号等分类细化

### 4. 视图实现依赖专门文档导览

`TRD` 负责决定：

- 视图文件怎么组织
- 关键区域、按钮、状态栏、继承点怎么实现
- 哪些视图行为要引用官方视图文档

具体视图规范查证时优先调用 `view-dev`。

### 5. 官方文档是基础必查

对于：

- ORM 用法
- 模型继承
- 安全规则
- 视图机制
- 数据文件和调度方式

都应优先查 `odoo18-docs` 或 `view-dev`，不要凭印象写技术结论。

### 6. 统一未决问题入口

所有尚未确认但会影响 `TRD` 结论的问题，统一进入：

- 当前业务域根页面下的页面 `待确认事项`

`TRD` 本身只保留：

- `关键假设`
- `已放弃 / 暂缓方案`
- `下一步重点确认`

### 7. 支持阶段性审查

当本次对 `TRD` 有实质写回，且准备交给 `dev` 使用时，`trd-v2` 应建议或按需调用 `review` 做快速审查。
