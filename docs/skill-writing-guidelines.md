# Odoo Forge Skill 写作规范

## 目标

这套 skill 系统不是为了“自动把文档写完”，而是为了让 AI 参与 Odoo 产品设计与开发时，能持续帮助团队把问题讨论清楚、把方案收敛正确、把共识沉淀进 FlowUS。

文档很重要，但文档不是目的。
真正重要的是中间那段高质量的共创过程：

- 把业务表述翻译成真正的问题
- 把模糊诉求拆成可讨论的块
- 把可选方案摆出来做权衡
- 把已经确认的共识沉淀下来，供下游继续使用

## 适用范围

本规范适用于 Odoo Forge 中所有面向 Odoo 产品设计、方案设计、技术设计、验收、查询、审查的 skill。

尤其适用于以下三类 skill：

- 共创产出类：`global-context`、`discovery`、`solution`、`prd`、`trd`、`uat`
- 审查类：`review`
- 顾问与工具类：`industry-practice`、`company-rules`、`odoo18-docs`、`module-research`、`module-report`、`view-spec`、`view-dev`、`translation`

## 核心原则

### 1. 不再依赖 Hub

不引入中央调度 skill。
每个 skill 都应该能被用户直接调用，或根据当前任务自然触发。

skill 结束时可以建议下一步，但不要承担强制路由职责。

### 2. Skill 必须无状态

skill 自身不维护项目状态机，不依赖“上一次对话记忆”才能继续工作。

每次运行都必须假设自己是“第一次介入”，并通过读取上游文档重建上下文。

外部记忆统一放在 FlowUS 文档中，而不是 skill 内部流程状态中。

### 3. 上游文档就是提示词工程

每个 skill 都必须明确自己的上游输入文档。

例如：

- `solution` 的上游是 `全局资源 + 域级 BRD`
- `prd` 的上游是 `全局资源 + 域级 BRD + 域级 Solution + 当前 epic 约束`
- `trd` 的上游是 `全局资源 + PRD + 域级 Solution + 相关 Odoo 参考`

skill 的第一步不是输出，而是先读这些文档，再恢复认知基线。

### 4. 讨论优先，文档其次

skill 不应把“生成文档”当成主流程。

正确的顺序应该是：

1. 先复述理解
2. 暴露空白与冲突
3. 分块讨论
4. 每块确认
5. 最后写回已经确认的内容

文档只是阶段性共识快照，不是 skill 的唯一价值。

### 5. 分步共创，不一口气写完

参考 `doc-coauthoring` 的思路，每个 skill 都应按块推进，而不是直接产出一整篇“看起来完整”的文档。

推荐节奏：

1. 当前阶段目标是什么
2. 这次先讨论哪一块
3. 抛出 1 个关键问题，或 2-3 个候选方向
4. 用户回应后，小结当前共识
5. 询问是否继续下一块

### 6. 建议下一步，不强制跳转

参考 `superpowers-main` 的做法，skill 在阶段结束后应该给出自然的下一步选项，例如：

- 继续补当前文档
- 进入下一个 skill
- 先做审查
- 先做探索原型

但这只是建议，不是强路由，不需要额外状态机。

### 7. 轻状态，重痕迹

不做复杂轮次管理，不依赖 `R0 / R1 / R2` 之类的外部状态机。

但是每份关键文档都应保留最小必要痕迹，至少包含：

- 当前共识
- 未决问题
- 关键假设
- 已放弃方案

这样既能保持轻量，又不会丢掉讨论过程的价值。

### 8. 始终顺着 Odoo 心智模型做抽象

skill 的职责不是照搬业务原话，而是帮助团队把业务原话翻译为更合理的产品能力表达。

例如：

- 业务说“要批量导入 Excel”
- skill 不能只把这句话原样写进文档
- 应该继续追问：这背后的真实问题是什么，Odoo 标准流程有没有更好的建档方式，导入究竟是主路径还是迁移/补录手段

## Skill 分类建议

### 共创产出类

职责是通过讨论把一个文档逐步收敛出来。

特点：

- 强依赖上游文档
- 强交互
- 一次只推进一个块
- 写回 FlowUS 的是确认后的内容

### 审查类

职责不是生成文档，而是作为横切质量门检查已有文档。

至少覆盖四类检查：

- 完整性
- 前后一致性
- Odoo 适配性
- 交付就绪度

### 顾问与工具类

职责是补充事实、参考和专业判断，不负责主导整篇文档。

这类 skill 的输出应尽量结构化，方便被共创 skill 或审查 skill 消费。

## 标准 Skill 流程

每个核心 skill 都应遵循下面这条主流程。

### 阶段 1：读取上下文

先读取必要的上游文档和参考资料，再明确三件事：

- 我已知什么
- 我还不知道什么
- 这次最适合先推进什么

如果上游文档缺失，skill 不要硬猜，应明确指出缺口，并建议先补哪份文档。

### 阶段 2：复述理解

开始正式讨论前，先用简洁语言向用户复述当前理解，包括：

- 本次处理的是哪个业务域 / EPIC
- 当前文档已经说清楚的部分
- 还模糊或冲突的地方
- 这次准备聚焦的块

这一步的目的不是礼貌，而是对齐认知基线。

### 阶段 3：分块共创

每次只推进一个块。

每个块推荐使用下面这个循环：

1. 说明本块目标
2. 提 1 个关键问题，或给出 2-3 个候选方案
3. 基于用户反馈继续追问或收敛
4. 输出本块小结
5. 询问是否进入下一块

适用示例：

- `discovery`：业务目标、角色、核心流程、异常流、规则约束、业务对象、跨域关系
- `solution`：方案路径选择、模块蓝图、核心模型、关键流程映射、GAP、原型范围
- `prd`：用户故事、状态机、规则、权限、视图、异常场景

### 阶段 4：写回确认内容

只有当一个块已经得到确认时，才把它写回 FlowUS 中对应的权威页面。

不要把下面这些东西当成已定稿写入正文：

- 仍在讨论的候选方案
- 纯猜测的结论
- 用户还没确认的结构化补全

未确认内容应进入：

- `待确认问题`
- `关键假设`
- `备选方案`

### 阶段 5：建议下一步

skill 结束时给出 2-4 个自然下一步选项。

示例：

- 继续完善当前文档
- 进入 `solution`
- 调 `review`
- 先做可丢弃的探索原型

## SKILL.md 推荐骨架

每个核心 skill 的 `SKILL.md` 推荐至少包含以下部分：

```md
---
name: <skill-name>
description: <触发描述，既写做什么，也写何时必须用>
---

# <标题>

## 你是谁
说明本 skill 的定位、价值、边界。

## 什么时候使用
明确触发场景、典型用户意图、输入前提。

## 必需上游上下文
列出必须读取的文档、缺失时的处理方式。

## 开始前先读什么
要求先读上游文档，再总结已有认知与缺口。

## 共创流程
按块推进，说明每块如何提问、如何收敛、如何确认。

## 写回规则
什么能写入正文，什么只能进入未决区。

## 输出结构
最终文档推荐结构，或结构化输出模板。

## 结束时给出的下一步建议
列出自然下一步，而不是强制路由。

## Guardrails
明确不能做什么、不要越界做什么。
```

## Description 写法要求

`description` 是最重要的触发器，必须同时说明：

- 这个 skill 能做什么
- 用户在什么语境下应该使用它
- 哪些关键词或任务类型必须触发它

写法建议：

- 准确但略“积极触发”
- 不要只写一个抽象名词
- 不要把触发条件藏在正文里

坏例子：

- `负责写 PRD`

好例子：

- `EPIC 级产品需求共创 skill。用于把已经确认的业务目标和域级方案逐步收敛为可交付 PRD。当用户提到“写 PRD”“整理功能规格”“把这个 epic 说清楚”“补规则/状态/权限/视图”时必须使用。`

## 正文写法要求

### 1. 解释为什么，不要只堆 MUST

可以明确要求，但更推荐说明“为什么这样做”，让 skill 更稳定。

### 2. 保持正文瘦身

`SKILL.md` 只放工作流、边界和关键模板。

大段参考内容应拆到：

- `references/`
- `assets/`
- `scripts/`

### 3. 变体内容放到 references

如果同一 skill 面向多个变体，应在正文中做选择，再按需读取对应参考文件。

### 4. 用例子约束输出

如果某种输出结构很关键，直接给模板或例子，不要只写“请输出结构化结果”。

### 5. 仓库模板只是写作辅助，不是最终落地位置

仓库中的模板、示例和方法说明，只用于指导 skill 如何组织内容并写入 FlowUS。

最终权威文档必须落在 FlowUS 页面中，而不是落在仓库里的 Markdown 文件中。

## FlowUS 文档约定

为了让 skill 真正无状态，关键文档建议统一包含这些区块：

- `当前共识`
- `待确认问题`
- `关键假设`
- `已放弃 / 暂缓方案`
- `下一步重点确认`

说明：

- `当前共识` 用于下游 skill 恢复当前已确认结论
- `待确认问题` 用于后续继续追问
- `关键假设` 用于标记暂时接受但未验证的判断
- `已放弃 / 暂缓方案` 用于保存已讨论但当前不采用的路径
- `下一步重点确认` 用于提示下次最值得推进的点

## FlowUS 页面定位与读取规则

所有正式文档都以 FlowUS 为唯一落地和读取来源。
不要把正式业务文档维护在本地文件中，也不要假设本地副本是最新真相。

### 顶层入口

所有 skill 进入项目文档体系时，都从顶层页面 `产品设计` 开始。

标准读取顺序：

1. 先定位标题为 `产品设计` 的根页面
2. 对根页面执行一次页面树读取，恢复当前目录结构
3. 再根据当前任务进入 `全局资源`、目标业务域或目标 EPIC

### 标准层级

当前约定的层级是：

- `产品设计`
- `全局资源`
- 各业务域
- 各业务域下的域级文档和 EPIC
- EPIC 下的 `PRD / TRD / UAT`

推荐理解方式：

- `产品设计`：整个产品设计空间的唯一根入口
- `全局资源`：所有业务域共享的项目级上下文
- 业务域页面：某个业务域的总入口，例如 `项目管理`、`经营管理`
- 域级文档：该业务域根页面下的 `BRD`、`Solution`、`待确认事项` 或其他域级认知文档
- EPIC 页面：该业务域下的交付拆分单元
- EPIC 子文档：`PRD`、`TRD`、`UAT`

### 所有 skill 的标准读取动作

所有核心 skill 都应按下面这个顺序读取：

1. 定位并读取 `产品设计` 页面树
2. 读取 `全局资源` 及其子页面
3. 定位目标业务域页面
4. 如果任务是域级共创或域级方案，先读该业务域根页面下的域级文档
5. 如果任务是 EPIC 级交付，再进入对应 EPIC 的 `PRD / TRD / UAT`

### 当域级文档未建立时

如果目标业务域根页面下还没有 `BRD`、`Solution` 或 `待确认事项`：

- 不要假装这些文档存在
- 明确告诉用户当前域级文档缺什么
- 先把现有业务域页面和已有 EPIC 页面读完，恢复当前上下文
- 如果缺少页面 `待确认事项`，先在当前业务域根页面下创建页面 `待确认事项`
- 再建议创建其他缺失的域级文档

### 当前项目的已知结构

当前已确认的 FlowUS 项目结构是：

- 根页面：`产品设计`
- 根页面下有：`全局资源`、`项目管理`、`经营管理`
- `全局资源` 下目前有：`项目背景`、`组织架构`
- `项目管理` 下目前已有多个 EPIC 页面，每个 EPIC 下有 `PRD / TRD / UAT`

这说明：

- 全局入口已经存在
- 全局资源已经存在
- 业务域页面已经存在
- 目前至少 `项目管理` 域已经进入 EPIC 交付层
- 域级 `BRD / Solution / 待确认事项` 页面需要按后续规范逐步补齐

### 禁止做法

- 不要跳过 `产品设计` 根页，直接凭印象猜某个页面路径
- 不要把 FlowUS 文档真相缓存成长期本地副本
- 不要在没有读取页面树的情况下假定某个业务域或 EPIC 一定存在
- 不要把域级文档和 EPIC 级文档混着读、混着写

## 页面《待确认事项》约定

所有 skill 都使用“当前作用域根页面”下的一份页面 `待确认事项`，用来承接“现在还不能进入正文、但需要产品或业务确认”的问题。

作用域根页面的约定如下：

- `global-context`：使用页面 `全局资源` 作为作用域根页面
- 域级共创、方案、交付类 skill：使用当前业务域根页面作为作用域根页面

也就是说：

- `global-context` 的未决问题写进 `产品设计 / 全局资源 / 待确认事项`
- `discovery`、`solution`、`prd`、`trd`、`uat` 的未决问题写进“当前业务域根页面 / 待确认事项”

不为每个 skill 单独维护未决清单，不做状态机，不做额外数据库，也不把它落到本地 Markdown 文件。

### 什么时候写入

只有在以下情况才写入页面 `待确认事项`：

- 该问题会影响当前文档的关键结论
- 需要产品、业务或上游信息确认
- skill 继续查资料也无法自行确认

以下情况不要写入：

- 只是措辞不够好
- 纯开发内部待办
- skill 自己继续查文档就能解决的问题
- 不影响结论的小疑问

### 记录格式

统一使用 Markdown checkbox 形式：

- 未确认：`- [ ]`
- 已确认并已回写权威文档：`- [x]`

每条事项只写一个明确问题，避免把多个问题揉在一起。

推荐格式：

```md
- [ ] `来源skill` 明确的问题句
  影响：影响哪份文档或哪个关键块
  当前判断：如果存在临时判断就写，没有就省略
```

示例：

```md
- [ ] `discovery` 设计变更申请被驳回后，是否允许原申请修改后再次提交？
  影响：BRD 核心流程、后续 PRD 状态流转

- [ ] `solution` 资质到期提醒应以站内待办为主，还是邮件通知为主？
  影响：Solution 模块蓝图
  当前判断：倾向站内待办为主，邮件为辅
```

### 关闭规则

当某个问题被产品或业务确认后：

1. 先更新对应的权威文档
2. 再把《待确认事项》中的对应条目标记为 `- [x]`

不要先勾选，再等以后补文档。
勾选动作表示“结论已经确认且已回写”。

### 每个 skill 中都要写的统一规则

每个核心 skill 都应包含一个《待确认事项处理》小节，至少表达以下意思：

- 当发现结论暂时不能进入正文时，不要硬写成已确认结论
- 先判断是否需要产品或业务确认
- 如果需要，主动询问用户是否加入页面 `待确认事项`
- 写入时使用统一 checkbox 格式
- 问题确认后，先更新权威文档，再勾选对应事项

## 所有 Skill 的共同规范

除了各自职责不同之外，所有 skill 都应共同遵守以下规范。

### 1. 先读后写

skill 开始后，先读自己的上游文档和必要参考，再开始输出。
不要跳过上下文恢复直接生成内容。

### 2. 先复述理解再推进

在进入实质讨论或产出前，先复述：

- 当前处理对象
- 已知共识
- 还不清楚的地方
- 这次准备推进的块

### 3. 只写确认过的结论

未确认、纯猜测、仍在讨论中的内容，不进入正文共识区。
这类内容只能进入：

- `关键假设`
- `已放弃 / 暂缓方案`
- 《待确认事项》

### 4. 不越权替代别的 skill

每个 skill 都要守住边界。

例如：

- `discovery` 不替代 `solution`
- `solution` 不替代 `prd`
- `review` 不替代主文档共创

### 5. 能追问就不要偷补

当关键信息缺失时，优先追问用户或引导用户选择，不要为了把文档写满而自行脑补。

### 6. 建议下一步，但不强制路由

skill 结束时给出自然下一步建议即可，不承担中心调度职责。

### 7. 输出要能被下游消费

skill 的输出应尽量结构化、清晰、可追溯，方便下游 skill 继续读取和使用。

### 8. 用统一术语

同一个业务对象、角色、状态、文档名，在不同 skill 中尽量保持同一命名，避免同义词漂移。

### 9. 面向真实问题，不面向表面诉求

skill 应帮助团队从业务原话中识别真实问题，而不是机械照抄表面需求。

### 10. 结束时说明本次改了什么

如果本次运行涉及文档更新，skill 应清楚说明：

- 新增了什么
- 修改了什么
- 哪些问题仍未决

## 域级与 EPIC 级边界

### 域级文档

域级文档是“认知收敛文档”，是活文档，可以持续完善。

- `discovery / BRD`：业务问题、业务流程、角色、规则、对象、跨域关系
- `solution`：Odoo 解法蓝图、模块组合、核心模型、关键流程映射、GAP、原型范围

### EPIC 级文档

EPIC 级文档是“交付文档”，在真正进入开发前再做深做实。

- `PRD`：把业务与交互规格说透
- `TRD`：把技术实现说透
- `UAT`：把验收动作说透

## Review Skill 设计要求

`review` 是已经落地的横切 skill。

建议不要一开始拆成很多 review skill，而是先做一个通用 `review`，按文档类型切模式。

最小能力要求：

- 审 `BRD`：问题定义是否真实，流程/角色/规则/边界是否完整
- 审 `solution`：是否顺着 Odoo 做，是否过度定制，模块边界是否合理
- 审 `PRD`：规则、状态、权限、视图、异常流是否可交付
- 审 `TRD`：模型、视图、安全、数据、测试指引是否足够开发
- 审 `UAT`：验收步骤是否可执行，主路径和关键异常是否覆盖

## Skill 作者自检清单

提交一个新 skill 或重写旧 skill 前，至少自检以下问题：

- 这个 skill 是否能在没有中央调度 skill 的前提下独立工作
- 它是否能只靠上游文档恢复上下文
- 它是否把“讨论过程”当作主流程，而不是直接生成全文
- 它是否支持按块共创，而不是一口气写完
- 它是否只把确认后的内容写回正文
- 它结束时是否会建议自然下一步
- 它是否明确了自己的边界，避免越权替代别的 skill
- 它的 `description` 是否足够容易被正确触发
- 它的正文是否足够精简，重内容是否已拆到 `references`
- 它是否至少准备了 2-3 个真实测试 prompt

## 落地顺序建议

建议按这个顺序改造现有 skill：

1. 先写统一规范
2. 先重写 `discovery` 作为样板
3. 再重写 `solution`
4. 然后统一 `prd / trd / uat`
5. 最后补齐并打磨 `review`

先把共创主链路做顺，再扩展工具和审查能力。
