# 模块搜索

## 定位

这是一个 Odoo 模块搜索 skill。

它只负责两件事：

1. 按需求搜索 OCA、第三方 GitHub 仓库和本地模块
2. 返回适合方案阶段快速判断的候选简表

它**不负责深度精读，也不写 FlowUS 报告库**。一旦用户已经选中模块、仓库或候选集合，需要转交给 `module-report`。

## 必需上游上下文

| 上游内容 | 必需性 | 用途 | 缺失时怎么做 |
|----------|--------|------|--------------|
| `全局资源` 页面及其子页 | 必需 | 了解项目背景，判断“推荐级别” | 如果读不到，明确说明推荐级别可信度下降 |
| 当前需求描述 / 方案块 / 业务问题 | 必需 | 决定搜索关键词和筛选方向 | 先要求调用方把需求说清楚，不要盲搜 |

## FlowUS 页面怎么找

如果调用方没有显式传入项目背景，默认从 FlowUS 恢复：

1. 先定位根页面 `产品设计`
2. 读取 `产品设计` 的页面树
3. 进入子页面 `全局资源`
4. 从 `项目背景`、`业务域全景图`、`项目约束` 中提炼项目特征

## 什么时候使用

以下场景必须使用本 skill：

- “帮我找一下 Odoo 有没有做 XX 的模块”
- “OCA 有没有现成模块能做这个需求”
- “帮我筛一下审批 / 合同 / 项目类模块，先给我个列表”
- “这个需求有哪些社区模块可选”
- `solution` 需要在官方能力之外补查 OCA / 第三方候选

以下情况不要用本 skill：

- 需要确认官方标准模块是否已经覆盖，用 `odoo18-docs`
- 需要深入精读某个已选模块，用 `module-report`
- 需要把模块报告沉淀进 FlowUS，用 `module-report`

## 默认假设

- 如果未明确 Odoo 版本，默认按 `18.0` 搜索和判断
- 来源不限：OCA、第三方 GitHub 仓库、本地路径都可纳入
- 本 skill 负责“找候选并给简表”，不替代官方能力查证

## 输入识别

| 输入形式 | 来源类型 | 处理方式 |
|---------|---------|---------|
| `OCA/server-tools`、`any-org/any-repo` | GitHub 仓库 | 读取仓库结构、搜索模块目录 |
| `~/code/addons/my_module`、`/opt/odoo/addons/...` | 本地路径 | 用文件系统读取 |
| “帮我找做审批的模块” | 搜索需求 | 联网搜索 + 仓库定位 + manifest 验证 |

自动判断，不需要用户手动指定来源。

## 工作流

1. **理解需求**
   提炼：
   - 业务对象
   - 核心动作
   - 关键词
   - 约束条件
   - 目标 Odoo 版本
2. **构造搜索线索**
   优先组合：
   - 业务关键词
   - 典型对象名
   - 常见 OCA 仓库映射
   - 当前项目场景的限制条件
3. **搜索候选**
   在 OCA、第三方 GitHub、本地路径中搜索相关模块。
4. **快速验证**
   至少读取：
   - `__manifest__.py`
   - `README*`
   必要时补看目录结构。
5. **初筛与分层**
   从下列维度做快速判断：
   - 功能是否贴近需求
   - 版本/分支是否匹配
   - 简介是否能解释模块定位
   - 对你们当前项目是否贴场景
6. **输出结果**
   给出候选简表，并说明哪些值得继续交给 `module-report`。

## 搜索与初筛规则

- 优先搜 OCA，再补第三方。
- 不确定归属时，用关键词直接搜，不要猜。
- 搜索结果必须至少验证 manifest，不能只凭仓库名判断。
- 推荐等级至少分为：
  - `推荐`
  - `备选`
  - `谨慎`
- 初筛时重点看这些信号：
  - 是否有对应版本分支
  - manifest 描述是否贴需求
  - 模块定位是否顺 Odoo 心智模型
  - 模块是否贴合你们当前项目场景

`推荐级别` 的判断必须结合 FlowUS 中的 `全局资源`，不能脱离项目背景空评。

## 输出格式

```markdown
搜索主题：{需求关键词} | 版本：{version}

| 模块名 | 简介 | 推荐级别 | 适用场景 |
|--------|------|----------|----------|
| {module_name} (`{org}/{repo_name}`) | {一句话简介} | 推荐 | {适合你们在哪类需求里考虑} |
| {module_name} (`{org}/{repo_name}`) | {一句话简介} | 备选 | {适合在哪些条件下作为备选} |
```

表格后只补三类短说明：

- 为什么推荐 / 备选 / 谨慎
- 是否值得继续做 `module-report`
- 如需继续精读，优先看哪个仓库 / 模块

## 找不到时怎么答

如果没有找到直接匹配模块，明确说明：

- 目前未找到直接匹配模块
- 是完全没有，还是只有弱相关候选
- 建议下一步是：
  - 回官方能力再确认
  - 缩小需求
  - 准备轻定制 / 新建模块

不要编造候选。

## Guardrails

- 不要把“搜到了相关关键词”当成“可用”
- 不要写 FlowUS 报告页或多维表
- 不要替代 `odoo18-docs` 做官方能力查证
- 不要替代 `module-report` 做正式模块报告
- 不要在没读 manifest 和 README 的情况下下推荐结论
