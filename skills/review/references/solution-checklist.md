# Solution 审查清单

## 必须检查

- 是否先查官方能力，再考虑 OCA / 社区，再考虑轻定制
- 模块与能力蓝图是否清楚
- 核心流程映射是否清楚
- GAP 是否明确
- 关键取舍是否说清楚
- 与 BRD 是否一致

## 高价值检查

- 是否顺着 Odoo 心智模型做设计
- 是否存在明显过度定制
- 是否把应该调整业务流程的问题，直接变成系统重定制
- 是否已经说明哪些能力来自官方、哪些来自 OCA、哪些必须定制

## 模块与证据检查

- 模块结论是否有官方能力依据
- 社区模块推荐是否经过搜索或精读支撑
- 关键定制点是否给出必要理由

## 红旗信号

- 方案看起来完整，但没有证据来源
- 方案大量依赖定制，却没有解释为什么不能顺标准流程
- GAP 只写“需定制”，没有解释范围和原因
- 模块清单很多，但没有取舍逻辑

## 适合加入页面《待确认事项》的典型问题

- 某个关键流程是否必须维持当前业务做法
- 某个社区模块是否真允许进入正式方案
- 某个重定制点是否可以接受业务调整后降级
- 某个制度或行业要求是否会改变方案路径
