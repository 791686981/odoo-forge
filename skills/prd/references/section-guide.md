# PRD 章节深度标准

本文件用于约束 `prd` 在各章节里“要写到多深才算够用”。

目标不是把文档写得很长，而是让 `trd` 和 `uat` 读完后不用再猜。

## 一、用户故事与范围

### 先总后分的做法

先列 EPIC 的能力总览或闭环清单，再拆用户故事。

不要一开始就零散写故事，结果最后才发现边界没圈住。

### 每条用户故事至少要满足

- 有明确角色，不要只写“用户”
- 有具体操作，不要只写“管理”“维护”
- 有可观察结果，要能看到状态、数据、通知或页面变化
- 有验收标准，不能只写“支持”
- 有来源，尽量标明来自 `BRD`、`Solution`、EPIC 概述或 `GAP-N`

推荐格式：

```markdown
| ID | 角色 | 操作 | 期望结果 | 验收标准 | 来源 |
|----|------|------|----------|----------|------|
| US-01 | {角色} | {操作} | {结果} | {标准} | {来源} |
```

### 范围排除必须写清楚

`PRD` 不是只写“做什么”，还要写“不做什么”。

至少回答：

- 哪些能力明确在本 EPIC 内
- 哪些能力明确不在本 EPIC 内
- 不做的原因是什么
- 如果归到其他 EPIC，要写清归属

## 二、业务对象与信息

### 先列对象总览，再展开重点对象

如果对象较多，先列一张对象总览表，再逐个展开主对象和关键从属对象。

### 怎么写才够用

- 写业务含义，不直接写技术实现
- 写关键信息项，不要只写对象名
- 当字段渲染方式会明显影响交互时，可以写 `渲染方式`
- 说明对象是新建还是在现有对象上扩展
- 如果对象或字段受 `Solution` 里的 `GAP` 影响，要显式标注来源

### 不够用的写法

- 只列字段名，没有说明
- 只有对象，没有对象关系
- 只写“支持录入”，没说录什么、为什么录

## 三、业务逻辑

这是 `PRD` 最核心的章节，默认拆成四层：

1. 主流程
2. 分支与异常处理
3. 状态机
4. 业务规则分类

### 1. 主流程

目标：

- 让人看懂 happy path 怎么走完
- 让 `trd` 知道每一步谁操作、系统做什么

推荐组合：

- Mermaid 图
- 步骤说明

当多角色顺序配合明显时，优先用 `sequenceDiagram`。
当分支决策多时，优先用 `flowchart`。

### 2. 分支与异常处理

至少覆盖：

- 取消
- 驳回
- 退回重提
- 超时
- 无数据或缺数据时的处理
- 例外权限

不要只有主流程，没有异常路径。

### 3. 状态机

当对象存在阶段性流转时，尽量补状态图和状态转换表。

状态转换表至少写：

- 起始状态
- 目标状态
- 触发操作
- 操作角色
- 前置条件
- 系统动作

### 4. 业务规则分类

推荐至少分成下面四类：

- 校验规则
- 自动计算
- 自动化动作
- 编号 / 命名规则

如果有审批、锁定、例外授权，也可补充对应规则表。

## 四、页面与菜单

### PRD 在页面章节里要回答什么

- 这个 EPIC 需要哪些视图
- 主工作界面是什么
- 关键操作从哪里进入
- 菜单层级怎么组织

`PRD` 不需要写到 XML 或技术属性层，但不能只写一句“有个表单页”。

### 什么时候调用 `view-spec`

以下情况优先调用 `view-spec`：

- 有状态机或明显工作流
- 有多个头部按钮
- 有明细行、统计或复杂表单结构
- 这是 EPIC 的主工作界面

以下情况可以只写简洁版说明：

- 纯 CRUD 配置页
- 字段很少的简单从属对象
- 只读统计页

## 五、角色与权限

权限要用业务语言写，让产品、业务和研发都看得懂。

推荐至少回答：

- 谁能看到哪些数据
- 谁能创建
- 谁能编辑
- 谁能删除
- 谁能推动状态变化
- 谁在什么情况下有例外权限

不要直接写技术 domain，也不要只写“按角色控制”。

## 六、按需章节

### 外部接口

只有和第三方系统有明确交互时才写。

### 报表与打印

当 EPIC 明确包含统计、透视、图表、打印单据时再写。

### 配置项

当存在可以影响业务行为的可配置开关、阈值或策略时再写。

没有内容就不要硬保留空章节。

## 七、常用辅助 skill

- `view-spec`
  - 用于复杂视图规格下钻
- `mermaid-diagrams`
  - 用于流程图、状态图、时序图
- `industry-practice`
  - 用于判断某些流程和规则在行业里通常做多严、多细
- `company-rules`
  - 用于审批、编号、归档、签署、质控、交付规范相关补充
- `review`
  - 当本次有实质写回，且准备交给下游时做快速审查
