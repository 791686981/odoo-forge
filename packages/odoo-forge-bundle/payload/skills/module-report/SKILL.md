# 模块报告

## 定位

这是一个 Odoo 模块报告 skill。

它只负责：

1. 深入阅读已选定的模块或仓库
2. 生成完整模块报告正文
3. 将模块报告沉淀到 FlowUS 的模块知识库

它**不负责做大范围候选搜索**。如果用户还没确定要看哪个模块，先交给 `module-research`。

## 必需上游上下文

| 上游内容 | 必需性 | 用途 | 缺失时怎么做 |
|----------|--------|------|--------------|
| `全局资源` 页面及其子页 | 必需 | 判断模块对你们项目的推荐度和使用场景 | 如果没有，仍可写报告，但要明确推荐度可信度下降 |
| 目标仓库 / 模块 / 本地路径 | 必需 | 决定要精读什么 | 没有目标就停止，先让调用方回到 `module-research` |
| `项目Wiki / 模块报告` 页面 | 建议 | 写入 FlowUS 模块知识库 | 如果没有，则先创建 |

## FlowUS 页面怎么找

写入时默认按下面的顺序定位：

1. 先定位根页面 `项目Wiki`
2. 读取 `项目Wiki` 的页面树
3. 在 `项目Wiki` 下进入或创建页面 `模块报告`
4. 在 `模块报告` 下按仓库名定位或创建多维表，多维表名字就是仓库名字
5. 在对应多维表内按 `标题` 查找记录；有则更新，没有则新建

创建或更新页面、多维表和模块记录时，遵循统一命名和 `icon` 约定：

- 页面标题保持纯文本，不把 emoji 写进标题
- `项目Wiki` 使用 `📚`
- 页面 `模块报告` 使用 `📦`
- 仓库级多维表使用 `🗃️`
- 模块记录页使用 `🧩`

## FlowUS 多维表约定

每个仓库一个多维表，表名直接使用仓库名字，例如：

- `server-tools`
- `project`
- `hr`

每个多维表只保留 4 个核心列：

- `标题`
- `简介`
- `使用场景`
- `推荐度`

其中：

- `标题` 使用标题列
- `简介` 使用文本字段
- `使用场景` 使用文本字段
- `推荐度` 使用 `select` 单选字段，选项固定为 `推荐 / 备选 / 谨慎`

点开每一行的页面后，正文写入完整模块报告。

## 什么时候使用

以下场景必须使用本 skill：

- “帮我精读这个模块”
- “帮我看看这个模块值不值得用”
- “把候选模块 A/B/C 各出一份精读结论”
- `solution` 已经找到候选模块，需要沉淀正式模块报告
- “把这个模块写进项目 Wiki 的模块报告库”

以下情况不要用本 skill：

- 只是想快速看看有哪些模块可选，用 `module-research`
- 只是想核实官方模块能力，用 `odoo18-docs`

## 工作流

1. 读 `__manifest__.py`
2. 读 `README*`
3. 读核心目录：
   - `models/`
   - `views/`
   - `security/`
   - `data/`
   - `controllers/`（如有）
4. 识别：
   - 核心模型
   - 关键视图 / 工作流
   - 依赖模块
   - 安装前置条件
   - 维护和兼容性信号
5. 结合 `全局资源` 我们的项目判断：
   - 对你们项目的推荐度
   - 使用场景
6. 生成完整模块报告正文
7. 写入 FlowUS：
   - 定位仓库名对应的多维表
   - 更新或创建对应模块这一行
   - 将完整报告写入该行页面正文
8. 返回简短摘要给调用方

## 正文标准

模块报告正文统一按下面 5 段写，不再自由发挥章节结构。

### 1. 模块概览

一句话说明模块做什么

### 2. 功能说明

详细介绍该模块解决什么问题,适用于什么场景

### 3. 依赖与安装

内容必须包含：

- 依赖模块
- 前置条件
- 安装步骤
- 安装后如何确认模块已正确安装

### 4. 如何配置

内容必须包含：

- 配置入口路径
- 配置步骤
- 关键配置项说明
- 推荐配置方式或常见配置值
- 配置完成后如何验证生效

### 5. 操作指南

这一段必须按“闭环场景”写，不要只罗列按钮或菜单。

每个闭环场景至少包含：

- 场景名称
- 适用角色
- 前置条件
- 操作步骤
- 预期结果

至少写 1 个完整闭环。
如果模块功能较多，可写 2-3 个典型闭环。

如果是多模块批量精读：

- 同仓库模块写入同一个多维表
- 不同仓库分别写入各自仓库名的多维表
- 每个模块独立占一行并拥有自己的正文页面

## FlowUS 写入结果

写入多维表时，摘要字段按下面的口径回填：

- `标题`：模块技术名或最常用名称
- `简介`：一句话说明模块做什么
- `使用场景`：适合在你们项目中的哪类需求里考虑
- `推荐度`：`推荐 / 备选 / 谨慎`

如果同一模块已经存在：

- 更新该模块对应行的字段
- 更新该行页面正文
- 不重复新建

## 质量标准

- 不只重复 README，要补代码层判断
- 不只说“有这个功能”，要说清楚模块的实际使用方式
- `推荐度` 必须结合 `全局资源` 判断，不能脱离项目背景
- `依赖与安装` 必须细到跟着就能装好
- `如何配置` 必须细到跟着就能配好
- `操作指南` 必须按闭环场景写，让用户能实际走通一遍
- 不确定的信息写“待确认”，不要编造

## 返回给调用方的摘要格式

写完 FlowUS 后，给调用方返回一段短摘要即可，例如：

```markdown
## 模块报告已更新

- 仓库：{repo_name}
- 模块：{module_name}
- 推荐度：推荐 / 备选 / 谨慎
- 使用场景：{一句话}
- 结论：{一句话总结}
```

## Guardrails

- 不要在用户还没选定目标时替代 `module-research`
- 不要再写本地 `docs/module_report/`
- 不要只看 README 就下结论
- 不要脱离 `全局资源` 直接给“推荐度”
- 不要为同一个模块重复创建多条记录
- 不要擅自新增这 5 段之外的强制章节
- 如果模块明显版本不匹配，要明确提示，不要硬推荐
