# 模块报告

## 定位

这是一个 Odoo 模块报告 skill。

它只负责：

1. 深入阅读已选定的模块或仓库
2. 生成完整模块报告正文
3. 将模块报告沉淀到 FlowUS 的模块知识库

它**不负责做大范围候选搜索**。如果用户还没确定要看哪个模块，先交给 `module-research`。

## 必需上游上下文

| 上游内容 | 必需性 | 用途 | 缺失时怎么做 |
|----------|--------|------|--------------|
| `全局资源` 页面及其子页 | 必需 | 判断模块对你们项目的推荐度、简介口径和使用场景 | 如果没有，仍可写报告，但要明确推荐度可信度下降 |
| 目标仓库 / 模块 / 本地路径 | 必需 | 决定要精读什么 | 没有目标就停止，先让调用方回到 `module-research` |
| `项目Wiki / 模块报告` 页面 | 建议 | 写入 FlowUS 模块知识库 | 如果没有，则先创建 |
| 相关业务域的 `BRD / Solution` | 推荐 | 结合当前业务域判断该模块在你们项目中的实际使用场景 | 如果没有，则至少基于 `全局资源` 做判断，并说明适用范围仍较宽 |

## 开始前先读什么

在进入模块源码之前，先用 FlowUS MCP 读取与你们项目有关的上下文，不要直接脱离项目背景写“推荐度 / 使用场景 / 简介”。

默认先读：

1. `产品设计 / 全局资源 / 项目背景`，了解项目范围、阶段和目标
2. `产品设计 / 全局资源 / 组织架构`，了解谁会用、谁维护、谁审批
3. `产品设计 / 全局资源 / 业务域全景图`，了解模块大概率落在哪些业务域
4. `产品设计 / 全局资源 / 术语表`，避免模块概览里的术语和你们项目口径打架
5. `产品设计 / 全局资源 / 现有系统清单`，判断是否涉及外部系统或现有系统边界
6. 如果模块是为某个明确业务域筛出来的，再读该业务域的 `BRD / Solution`

读这些内容的目的要明确：

- `简介`：写成适合你们项目口径的一句话，而不是照抄模块 README
- `使用场景`：写清楚在你们项目的哪类业务域、流程、角色或需求下可以考虑
- `推荐度`：结合你们项目背景判断是 `推荐 / 备选 / 谨慎`

## FlowUS 页面怎么找

写入时默认按下面的顺序定位：

1. 先定位根页面 `项目Wiki`
2. 读取 `项目Wiki` 的页面树
3. 在 `项目Wiki` 下进入或创建页面 `模块报告`
4. 在 `模块报告` 下按仓库名定位或创建多维表，多维表名字就是仓库名字
5. 在对应多维表内按 `标题` 查找记录；有则更新，没有则新建

如果对象不存在，按下面的 FlowUS MCP 模板创建，不要自由发挥。

### 页面与多维表创建模板

1. 创建 `项目Wiki` 页面：

```json
{
  "properties": {
    "title": "项目Wiki"
  },
  "icon": "📚"
}
```

2. 在 `项目Wiki` 下创建 `模块报告` 页面：

```json
{
  "parent": {
    "page_id": "{项目Wiki_page_id}"
  },
  "properties": {
    "title": "模块报告"
  },
  "icon": "📦"
}
```

3. 在 `模块报告` 页面下创建仓库级多维表。
这里必须创建成页面内可见的内嵌多维表，也就是 `is_inline: true`：

```json
{
  "parent": {
    "page_id": "{模块报告_page_id}"
  },
  "title": [
    {
      "type": "text",
      "text": {
        "content": "server-tools"
      }
    }
  ],
  "icon": {
    "emoji": "🗃️"
  },
  "is_inline": true,
  "properties": {
    "标题": { "title": {} },
    "简介": { "rich_text": {} },
    "使用场景": { "rich_text": {} },
    "推荐度": {
      "select": {
        "options": [
          { "name": "推荐" },
          { "name": "备选" },
          { "name": "谨慎" }
        ]
      }
    }
  }
}
```

4. 在仓库级多维表里创建模块记录页：

```json
{
  "parent": {
    "database_id": "{仓库级多维表_database_id}"
  },
  "properties": {
    "title": "sale_order_tag",
    "简介": "为销售订单提供标签分类与筛选能力",
    "使用场景": "适合在你们项目中需要按业务主题、项目来源或专项任务给销售订单分类时考虑",
    "推荐度": "备选"
  },
  "icon": "🧩"
}
```

### 创建后的验证步骤

每次创建仓库级多维表后，都要立刻验证它是不是“页面内可见”：

1. 对 `模块报告_page_id` 执行一次 `get_block_children`
2. 确认返回结果里出现刚创建的仓库级多维表
3. 如果页面里看不见，但数据库确实建出来了，优先怀疑创建成了 standalone database
4. 此时不要继续写记录，先删掉错误数据库，再按 `is_inline: true` 重建

不要只以“create_database 调用了成功”作为结束条件，要以“父页面能看到这个内嵌多维表”作为成功标准。

## FlowUS 多维表约定

每个仓库一个多维表，表名直接使用仓库名字，例如：

- `server-tools`
- `project`
- `hr`

每个多维表至少要有下面 4 个核心列，不能漏：

- `标题`
- `简介`
- `使用场景`
- `推荐度`

其中：

- `标题` 使用标题列
- `简介` 使用文本字段
- `使用场景` 使用文本字段
- `推荐度` 使用 `select` 单选字段，选项固定为 `推荐 / 备选 / 谨慎`

如果多维表不存在，创建时就把 4 个字段一次建齐，而且必须是 `is_inline: true` 的内嵌多维表。目标 schema 参考：

```json
{
  "标题": { "title": {} },
  "简介": { "rich_text": {} },
  "使用场景": { "rich_text": {} },
  "推荐度": {
    "select": {
      "options": [
        { "name": "推荐" },
        { "name": "备选" },
        { "name": "谨慎" }
      ]
    }
  }
}
```

如果多维表已经存在，但缺少 `简介 / 使用场景 / 推荐度` 任一列：

- 先补齐缺失列
- 再创建或更新模块记录

创建或更新模块记录时，`标题 / 简介 / 使用场景 / 推荐度` 这 4 个属性必须同时写，不允许只写 `标题`。

记录属性写入口径参考：

```json
{
  "title": "sale_order_tag",
  "简介": "为销售订单提供标签分类与筛选能力",
  "使用场景": "适合在你们项目中需要按业务主题、项目来源或专项任务给销售订单分类时考虑",
  "推荐度": "备选"
}
```

点开每一行的页面后，再写入完整模块报告正文。

## 什么时候使用

以下场景必须使用本 skill：

- “帮我精读这个模块”
- “帮我看看这个模块值不值得用”
- “把候选模块 A/B/C 各出一份精读结论”
- `solution` 已经找到候选模块，需要沉淀正式模块报告
- “把这个模块写进项目 Wiki 的模块报告库”

以下情况不要用本 skill：

- 只是想快速看看有哪些模块可选，用 `module-research`
- 只是想核实官方模块能力，用 `odoo18-docs`

## 工作流

1. 读 `__manifest__.py`
2. 读 `README*`
3. 读核心目录：
   - `models/`
   - `views/`
   - `security/`
   - `data/`
   - `controllers/`（如有）
4. 识别：
   - 核心模型
   - 关键视图 / 工作流
   - 依赖模块
   - 安装前置条件
   - 维护和兼容性信号
5. 再回到 FlowUS 上下文判断：
   - 对你们项目的推荐度
   - 在你们项目中的使用场景
   - 适合写进多维表的简介口径
6. 生成完整模块报告正文，固定按 5 段结构写
7. 写入 FlowUS：
   - 定位仓库名对应的多维表
   - 先确认多维表 4 个核心列齐全
   - 更新或创建对应模块这一行，并同时回填 `标题 / 简介 / 使用场景 / 推荐度`
   - 将完整报告写入该行页面正文
8. 返回简短摘要给调用方

## 正文标准

模块报告正文统一按下面 5 段写，不再自由发挥章节结构。

### 1. 模块概览

这一段至少要写清楚：

- 模块核心作用，一句话说清它解决什么问题
- 模块主要服务哪个业务对象、流程或工作台
- 模块来源和维护信号是否正常

不要只重复 README 第一段。

### 2. 功能说明

这一段重点写模块“实际做了什么”，至少要覆盖：

- 核心模型或关键业务对象
- 关键页面、动作或工作流
- 关键规则、自动化或联动
- 与标准 Odoo 相比的主要补充点

不要把“功能说明”和“使用场景”混写在一起。

### 3. 使用场景

这一段必须结合项目背景写，不能泛泛而谈。

至少回答这几个问题：

- 在你们项目的哪个业务域或哪类需求里可以考虑这个模块
- 更适合作为主方案、补充方案，还是备选方案
- 如果要落地，前提通常是什么

如果没有足够上下文支撑精确判断：

- 先明确是基于 `全局资源` 的哪部分判断
- 再说明哪些业务背景仍待确认

### 4. 依赖与安装

内容必须包含：

- 依赖模块
- 前置条件
- 安装步骤
- 安装后如何确认模块已正确安装

### 5. 操作指南

这一段必须同时包含“配置指南”和“一个演示场景”，不要只罗列按钮或菜单。

至少拆成这两块：

- `配置指南`
- `演示场景`

`配置指南` 至少包含：

- 配置入口路径
- 配置步骤
- 关键配置项说明
- 推荐配置方式或常见配置值
- 配置完成后如何验证生效

`演示场景` 至少包含：

- 场景名称
- 适用角色
- 前置条件
- 操作步骤
- 预期结果

至少写 1 个完整闭环演示场景。
如果模块功能较多，可写 2-3 个典型演示场景。

如果是多模块批量精读：

- 同仓库模块写入同一个多维表
- 不同仓库分别写入各自仓库名的多维表
- 每个模块独立占一行并拥有自己的正文页面

## FlowUS 写入结果

写入多维表时，摘要字段按下面的口径回填：

- `标题`：模块技术名或最常用名称
- `简介`：一句话说明模块做什么，语气与术语要尽量贴近你们项目
- `使用场景`：适合在你们项目中的哪类业务域、流程或需求里考虑
- `推荐度`：`推荐 / 备选 / 谨慎`

如果同一模块已经存在：

- 更新该模块对应行的 4 个字段
- 更新该行页面正文
- 不重复新建

## 质量标准

- 不只重复 README，要补代码层判断
- 不只说“有这个功能”，要说清楚模块的实际使用方式
- `简介 / 使用场景 / 推荐度` 必须结合 `全局资源`，有业务域上下文时还要结合对应 `BRD / Solution`
- 创建或更新模块记录时，不能漏掉 `简介 / 使用场景 / 推荐度`
- `依赖与安装` 必须细到跟着就能装好
- `操作指南` 里的配置指南必须细到跟着就能配好
- `操作指南` 必须包含至少 1 个闭环演示场景，让用户能实际走通一遍
- 不确定的信息写“待确认”，不要编造

## 返回给调用方的摘要格式

写完 FlowUS 后，给调用方返回一段短摘要即可，例如：

```markdown
## 模块报告已更新

- 仓库：{repo_name}
- 模块：{module_name}
- 简介：{一句话简介}
- 推荐度：推荐 / 备选 / 谨慎
- 使用场景：{一句话}
- 结论：{一句话总结}
```

## Guardrails

- 不要在用户还没选定目标时替代 `module-research`
- 不要再写本地 `docs/module_report/`
- 不要只看 README 就下结论
- 不要脱离 `全局资源` 直接给“推荐度”
- 不要为同一个模块重复创建多条记录
- 不要擅自新增这 5 段之外的强制章节
- 如果模块明显版本不匹配，要明确提示，不要硬推荐
