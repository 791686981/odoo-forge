# TRD 章节深度标准

本文件用于约束 `trd` 在各章节里“要写到多深才算够用”。

目标不是把文档写得很长，而是让研发或 AI 读完后可以直接进入实现。

## 一、模块规划

### 先总后分的做法

先确认本 TRD 的范围和模块拆分，再进入模型和方法细节。

不要一开始就零散写模型和方法，结果最后才发现模块边界混乱。

### 至少要回答

- 本 TRD 实现什么，不实现什么
- 需要几个模块
- 每个模块的职责是什么
- 模块之间是否有依赖或桥接关系
- `manifest` 需要承载什么
- 目录结构准备怎么组织

### 不够用的写法

- 只有模块名，没有职责
- 只有一个大模块，但没有说明为什么不拆
- 只有 `depends`，没有说明依赖目的

## 二、模型定义

### 先列模型来源，再展开重点模型

如果模型较多，先列模型来源对照表，再逐个展开核心模型和关键扩展模型。

### 怎么写才够用

- 说明模型是新建、继承扩展还是原生不改
- 字段至少写到能支持实现和视图
- 关键关系字段必须说清楚
- 当模型承担关键工作流职责时，方法入口也要标出
- 跨模型联动要显式标注

### 不够用的写法

- 只列模型名，没有字段和关系
- 只列字段，没有说明来源或职责
- 只说“继承某模型”，但不说加了什么

## 三、PRD -> TRD 映射

这是 `TRD` 最核心的约束之一。

### 必须做到

- `PRD` 中的关键规则、状态和自动化动作，在 `TRD` 中都能找到技术落点
- 每条关键规则至少有一种实现方式说明
- 对应的方法、文件或机制要能落到代码层

### 不够用的写法

- 直接写技术实现，但不回扣 `PRD`
- 只说“按 PRD 实现”，没有映射表

## 四、业务逻辑实现

推荐至少分成下面五类：

- 校验规则实现
- 自动计算实现
- 工作流实现
- 自动化动作实现
- 编号 / 命名规则实现

### 写到什么程度才够

- 方法名
- 所属模型
- 实现方式
- 触发时机
- 关键逻辑
- 对应哪条 `PRD` 规则

如果存在跨模型触发、定时任务、模板通知，也要明确指出入口和调用链。

## 五、权限

权限至少要回答：

- 有哪些角色组
- 每个组的层级关系是什么
- `ir.model.access.csv` 怎么配
- 是否需要记录规则
- 规则和 `PRD` 的角色边界是否一致

不要只写“按标准权限实现”。

## 六、视图实现

### TRD 在视图章节里要回答什么

- 需要哪些视图文件
- 关键视图区域怎么组织
- 哪些按钮、状态栏、明细区、`chatter`、动作需要技术说明
- 是否存在继承视图、`xpath`、`context`、`domain`

### 什么时候调用 `view-dev`

以下情况优先调用 `view-dev`：

- 继承视图较复杂
- `xpath` 和 `position` 不直观
- 按钮 / 动作 / `context` / `domain` 有技术风险
- 涉及特殊 widget、复杂搜索、看板、甘特等

## 七、数据文件

至少覆盖：

- 序号规则
- 通知模板
- 定时任务
- 配置 / 初始化数据

没有内容就不要硬保留空章节。

## 八、测试指引

测试指引不是 `UAT` 的重复，而是给研发的自测和逻辑验证清单。

推荐至少覆盖：

- 工作流
- 计算
- 约束
- 权限
- 序号或自动化

### 不够用的写法

- 只写“补测试”
- 只写“参考 UAT”

## 九、外部接口

只有 `PRD` 明确有外部接口需求时才写。

至少回答：

- 端点
- 认证
- 请求 / 响应结构
- 触发方式
- 失败处理

## 十、常用辅助 skill

- `odoo18-docs`
  - 用于 ORM、模型继承、安全、数据文件等官方机制查证
- `view-dev`
  - 用于视图机制、继承视图、动作、widget 查证
- `mermaid-diagrams`
  - 用于依赖图、模型关系图、联动链
- `company-rules`
  - 用于命名、编码、目录、规范边界补充
- `review`
  - 当本次有实质写回，且准备交给下游时做快速审查
