# Odoo 视图规格设计

## 定位

这个 skill 生成**完整视图规格** = ASCII 线框图 + 字段说明。

- **线框图** — 骨架级布局，产品确认结构和信息层次
- **字段说明** — 每个字段的 widget、备注；每个按钮的显示条件和行为

两者合在一起，AI 读了可以直接开发，不需要再猜。

不做的事：不写 XML 代码，不写 Python 逻辑，不讨论数据库设计。

---

## 调用模式

### 独立使用

用户直接要求设计视图时，走完整流程（确认范围→加载模板→输出）。

### 被产出专家调用（快速模式）

被 `prd` 或 `trd` 调用时，上下文信息已经齐了，**跳过提问，直接出图**：

| 调用方 | 已有上下文 | view-design 直接用 |
|--------|----------|-------------------|
| PRD > 页面与菜单 | PRD > 业务对象（字段清单）、PRD > 状态机、PRD > 角色权限 | 从业务对象提取字段→选骨架→画图 |
| TRD > 视图实现 | TRD > 模型定义（字段+类型）、TRD > 业务逻辑（工作流方法）、TRD > 权限 | 从模型定义提取字段→补充技术 attrs |

快速模式下：
- 不再问"有哪些字段"——从上下文提取
- 不再问"有没有状态机"——从上下文判断
- 直接选骨架、画图、输出

## 骨架选择标准

Form 视图有三种骨架变体，选择标准跟 PRD 的视图详细度判断对齐：

| 骨架 | 适用场景 | 复杂度指标 |
|------|---------|-----------|
| **简单 Form** | 配置项、简单记录、弹窗表单 | 无状态机、无明细行、字段 ≤ 8 个 |
| **标准 Form** | 普通业务单据 | 有明细行或多 Tab，状态 ≤ 3 个，按钮 ≤ 3 个 |
| **完整 Form** | 核心业务对象 | 状态机 ≥ 4 个状态、或按钮 ≥ 4 个、或明细行 + 多 Tab |

PRD 调用时的对应关系：
- PRD 说"需要详细线框图" → 选**完整 Form**
- PRD 说"简洁描述即可" → 选**简单 Form**，可以只出字段说明表不画线框

## 工作流程

### 第一步：确认范围

了解以下信息（可直接从上下文推断，不必逐一追问）：

1. 要设计哪几个视图？（Form + List + Search / + Kanban 等）
2. 是新建模型还是继承已有模型（如 sale.order）？
3. 主要字段有哪些？有状态机 / 工作流吗？

### 第二步：加载模板

1. 读 `references/layout-rules.md` 了解布局规则和 widget 视觉形态
2. 读对应视图的 `references/*-templates.md`
3. 按骨架选择标准选变体

### 第三步：输出视图规格

每个视图都包含线框图 + 说明表，格式如下：

**Form 视图：**
1. ASCII 线框图
2. 字段说明表：`字段 | Widget | 备注`（含可编辑条件、默认值、计算说明）
3. 操作按钮表：`按钮 | 显示条件 | 行为`
4. 状态栏说明（如有工作流）

**List 视图：**
1. ASCII 线框图（表头 + 示意数据行）
2. 列说明表：`列 | Widget | 备注`（含排序、聚合）

**Kanban 视图：**
1. 单张卡片 ASCII 线框图（不需要画所有列）
2. 卡片字段说明表：`字段 | Widget | 位置（主体/footer）`
3. 列/阶段说明（文字，不需要画出所有列）

**Search 视图：**
1. ASCII 线框图（搜索栏 + 三列下拉面板）
2. 筛选器表：`标签 | 过滤条件 | 所在组`
3. 分组依据表：`标签 | 字段`

**Pivot / Graph / Calendar 视图：**
1. 文字描述（这类视图结构简单，不需要 ASCII 线框图）
2. 字段/度量说明表

**多视图时：** 先出视图组合决策，再逐个视图输出，最后加菜单结构。

## 继承已有视图

实际项目中大量 EPIC 是扩展已有模型（sale.order、purchase.order、stock.picking 等），不是从零新建。继承视图的原则：**只描述变更，不重画整个视图。**

### 继承视图输出格式

```markdown
### 继承：sale.order Form 视图

**基础视图来源：** sale.view_order_form（Odoo 标准销售订单表单）

#### 变更清单

| 操作 | 参考位置 | 内容 | 说明 |
|------|----------|------|------|
| 新增字段 | partner_id 之后 | approval_user_id | widget=many2one_avatar，标签"审批人" |
| 新增按钮 | header，[确认订单] 之前 | [提交审批] | invisible="state != 'draft'"，action_submit_approval |
| 新增状态值 | state 字段 | 'approval' | statusbar 新增"审批中"节点 |
| 新增 Tab | notebook 末尾 | [审批记录] | 含 approval_line_ids inline list |
| 修改字段 | amount_total | 增加 decoration | decoration-bf="amount_total > 100000" |
| 隐藏字段 | payment_term_id | 条件隐藏 | invisible="state == 'approval'" |

#### 新增 Tab 线框图（仅画新增部分）

（如果新增内容复杂，可以单独画线框图）

```

### 继承视图要点

| 要点 | 说明 |
|------|------|
| 参考位置 | 写"谁的前面/后面"，不需要写 xpath |
| 只画新增 | 不重画整个视图，只画新增的 Tab、新增的字段组 |
| 标注基础视图 | 写清楚继承的是哪个 xml_id |
| 状态值扩展 | 如果在已有 state 字段上加新状态，单独说明 |
| 按钮位置 | header 按钮注明在哪个按钮之前/之后 |

### 继承场景的视图组合

继承场景通常不需要全套视图。常见模式：

| 场景 | 通常需要继承的视图 |
|------|-------------------|
| 在已有模型上加几个字段 | Form + List（加列）+ Search（加筛选） |
| 在已有模型上加工作流 | Form（加按钮+状态）+ List（加状态列） |
| 在已有模型上加明细行 Tab | 只改 Form（加 Tab） |
| 新建关联模型（如审批记录） | 新模型全套 + 主模型 Form 继承（加 Tab） |

## ASCII 画法约定

### 符号规则

```
┌─────┐ └─────┘ ├─────┤   边框和分隔线
│     │                     竖线边框
[按钮名]                    按钮
○ → ○ → ●                  状态栏（○ 未到达，● 当前）
☐ / ☑                      复选框（未选 / 已选）
[👤]                        头像
▼ / ▶                       折叠展开
≡                           拖拽手柄
```

### 标注规则

线框图里同时出现两种标注：
- **业务标注**（产品看）— 中文字段名，如"客户名称"、"合同金额"
- **结构标注**（开发看）— `── 区域名 ──` 标在边框上，如 `── header ──`、`── notebook ──`

```
┌── header ────────────────────────────────┐   ← 开发看
│ [提交审批] [批准]      ○ 草稿 → ○ → ●  │   ← 产品看
├── sheet ─────────────────────────────────┤
│  合同编号    HTxxx-001                   │
│  客户名称    XXXXX                        │
└──────────────────────────────────────────┘
```

## 多视图场景

一个业务对象通常需要多个视图配合，先决定组合再逐个输出：

| 场景 | 推荐视图组合 |
|------|-------------|
| 标准 CRUD | Form + List + Search |
| 有工作流 / 需看进度分布 | Form + List + Kanban + Search |
| 有统计/报表需求 | 上述 + Pivot / Graph |
| 有日程/排期需求 | 上述 + Calendar |
| 系统配置 | Settings（单独） |
| 只读统计 | List + Search + Pivot + Graph（只读 Form 或无 Form）|
