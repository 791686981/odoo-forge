# 全局资源页面规格

本文件用于指导 `global-context` 如何在 Notion 的 `全局资源` 下建设和维护项目级页面。

它是共创时的覆盖检查框架，不是必须逐条照念的问题清单。

## 使用原则

- 最终权威内容写回 Notion 的页面，不写回本地 Markdown
- 一次只推进一页或一页中的一个块
- 先保住“够用”，再追求“完整”
- 未确认的问题进入 `全局资源` 下的页面 `待确认事项`

## 页面优先级

| 优先级 | 页面 | 作用 | 最适合参与的人 |
|--------|------|------|----------------|
| P0 | 项目背景 | 说明为什么做这个项目、做多大、做到哪里算成 | 项目负责人 / PO / 团队负责人 |
| P0 | 组织架构 | 明确组织、职责和项目参与角色 | 项目负责人 / HR / 业务负责人 |
| P0 | 业务域全景图 | 给后续域级调研提供业务边界和顺序 | PO / 团队负责人 |
| P1 | 术语表 | 统一行业术语、内部缩写和易混概念 | PO / 行业顾问 / 核心业务 |
| P1 | 现有系统清单 | 明确现状工具、系统边界和迁移压力 | PO / IT / 核心业务 |
| P1 | 技术架构概览 | 明确项目级技术底线和实现策略 | 技术负责人 |
| P2 | 项目约束 | 明确时间、人力、预算、合规和风险边界 | 项目负责人 / 团队负责人 |
| P2 | 决策记录 | 沉淀项目中的关键判断，减少反复争论 | 项目负责人 / 团队负责人 |

## 项目背景

### 这一页解决什么问题

让任何后续 skill 都能快速知道：

- 这是什么项目
- 为什么现在要做
- 目标做到什么程度
- 明确不做什么

### 至少要写到什么程度

至少能回答下面四个问题：

- 公司或组织大致是什么类型、处在什么行业
- 项目启动的主要驱动力是什么
- 一期或当前阶段的目标是什么
- 明确不在当前范围内的内容是什么

### 典型红旗

- 只有“提升效率”“实现数字化”这种口号
- 完全没有范围边界
- 只写“业务提了很多需求”，没有项目目标

### 建议块

- 项目概述
- 启动背景与痛点
- 当前阶段目标
- 范围边界
- 关键里程碑

### 引导方式

- “这个项目为什么现在一定要做？”
- “做成之后，你们希望先解决哪几件最痛的事？”
- “有什么明确不在这次范围内？”

## 组织架构

### 这一页解决什么问题

让后续 skill 看懂谁是核心用户、谁拍板、谁协同、谁会被影响。

### 至少要写到什么程度

至少要说清楚：

- 主要部门和基本职责
- 关键岗位和汇报关系
- 各角色在项目中的参与方式

### 典型红旗

- 只有部门名字，没有职责
- 只知道“业务很多人会用”，不知道谁是关键人
- 项目角色和组织角色混在一起

### 建议块

- 组织概览
- 核心部门与职责
- 项目参与角色
- 汇报关系或协作关系图

### 可视化建议

组织关系适合用 Mermaid 图，但图只是辅助，文字职责不能省。

## 业务域全景图

### 这一页解决什么问题

让团队知道整个系统由哪些业务域构成、域之间怎么衔接、先后顺序大致如何。

### 至少要写到什么程度

至少要说清楚：

- 主要业务域列表
- 每个域的一句话职责
- 域之间的上下游关系
- 如果已有优先级，写清先后顺序

### 典型红旗

- 业务域只是按菜单名罗列
- 没有上下游关系
- 把 EPIC 当成业务域

### 建议块

- 业务域清单
- 每个域的一句话描述
- 上下游关系
- 当前阶段优先级

## 术语表

### 这一页解决什么问题

减少产品、业务、研发之间因为术语理解不一致造成的偏差。

### 至少要写到什么程度

先收录最常出现、最容易误解的术语即可，不要求首版就很全。

### 典型红旗

- 不同人用同一个词表达不同意思
- 术语很多，但没有统一解释
- 内部缩写没人敢确认含义

### 建议块

- 行业术语
- 公司内部叫法
- 易混概念辨析

### 推荐格式

```md
| 术语 | 含义 |
|------|------|
| EPC | Engineering, Procurement, Construction |
```

## 现有系统清单

### 这一页解决什么问题

帮助后续判断：

- 现在到底有哪些工具或系统在支撑业务
- 哪些是要保留、替换还是只做参考
- Excel、旧系统、纸质流程分别压在哪里

### 至少要写到什么程度

至少要覆盖：

- 系统或工具名称
- 主要用途
- 使用部门
- 当前状态

### 典型红旗

- 只知道“大家都在用 Excel”，不知道具体哪张表
- 不清楚哪些旧系统必须保留
- 没有说明现有数据在哪里

### 建议块

- 系统/工具列表
- 使用对象与用途
- 去留判断
- 数据形态与接口情况

## 技术架构概览

### 这一页解决什么问题

给后续方案设计和研发一个项目级技术底线，而不是详细实现方案。

### 至少要写到什么程度

至少应明确：

- Odoo 版本和部署大方向
- 官方 / 社区 / 定制的大原则
- 已知的技术约束或安全要求
- 团队级开发规范和基础设施约束

### 典型红旗

- 只有“我们用 Odoo 18”这一句
- 把模块设计和视图实现细节提前写进来
- 没有人能回答部署和安全边界

### 建议块

- 技术栈与部署方式
- 选型原则
- 已知核心模块或能力边界
- 安全、性能、合规约束
- 团队开发约定

### 提醒

这页最好有技术负责人参与。
如果当前无法完全确认，先写已知部分，再把关键问题放进页面 `待确认事项`。

## 项目约束

### 这一页解决什么问题

告诉后续 skill：这个项目有哪些真实限制，哪些判断不能脱离这些限制去做。

### 至少要写到什么程度

至少要覆盖：

- 时间约束
- 人力约束
- 预算或资源约束
- 合规或监管约束
- 关键风险

### 典型红旗

- 只知道要“尽快上线”
- 没有写团队规模和能力边界
- 没把外部监管要求记下来

### 建议块

- 时间计划
- 团队与资源
- 预算与采购约束
- 合规要求
- 已知风险

## 决策记录

### 这一页解决什么问题

把已经拍板的重要判断记下来，减少后面反复讨论。

### 至少要写到什么程度

不需要补历史大全。
只要把对后续影响明显的关键决策记录下来即可。

### 典型红旗

- 团队里每个人记得的“之前说好过的事”都不一样
- 范围调整过，但没有留下痕迹
- 某个重要判断为什么做出已经说不清楚

### 推荐格式

```md
| 日期 | 决策 | 原因 |
|------|------|------|
| 2026-04 | 一期先做项目管理域 | 该域痛点最集中，且能最快形成演示闭环 |
```

## 页面《待确认事项》

### 这一页解决什么问题

承接当前还不能进入项目级正文、但需要进一步确认的问题。

### 什么时候该写

- 会影响项目级页面的关键结论
- 需要项目负责人、产品、业务或技术负责人确认
- skill 自己继续查资料也无法确认

### 什么时候不要写

- 只是措辞修改
- 纯开发内部待办
- 业务域细节，但不影响项目级判断

### 推荐格式

```md
- [ ] `global-context` 设计院是否需要把分子公司作为独立组织单元纳入一期范围？
  影响：组织架构、业务域边界、权限策略
```
