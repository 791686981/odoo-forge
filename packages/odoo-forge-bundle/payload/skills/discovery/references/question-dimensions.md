# Discovery V2 深度调研框架

这不是固定顺序的问卷，而是一个覆盖检查框架。

使用方式：

- 先判断当前最值得推进的块
- 只在该块不够清楚时，打开对应维度继续深挖
- 每个维度都要追到“足以支持后续 solution”的清晰度

---

## 维度 1：业务目标与价值

### 这一维要搞清楚什么

- 这个业务域为什么存在问题
- 为什么现在必须处理
- 成功之后业务真正会变好什么

### 什么程度才算问透

- 能说清楚当前做法是什么
- 能说清楚最痛的 2-3 个问题
- 能说清楚谁因为这些问题受影响
- 能说清楚至少 1-3 个可验证的成功指标

### 红旗信号

- 目标只有“提升效率”“规范管理”“数据统一”
- 说不清现在到底怎么做
- 说不清不做会怎样
- 没有具体成功标准

### 追问方向

- 现状是 Excel、聊天、纸质单据还是旧系统
- 最痛的环节发生在哪一步
- 出错时对项目、成本、进度、责任有什么影响
- 哪些人最希望这件事先解决
- 成功之后如何量化

### 本块至少要沉淀到 BRD 的内容

- 当前做法
- 核心痛点
- 业务目标
- 成功指标

---

## 维度 2：干系人与用户角色

### 这一维要搞清楚什么

- 哪些部门、岗位、外部角色参与这个域
- 各角色分别做什么、关心什么、承担什么责任

### 什么程度才算问透

- 能区分决策者、使用者、审批者、被通知者、外部协作方
- 能说清每个关键角色在流程中的动作
- 能说清哪些角色只看、哪些角色要录入、哪些角色要审批

### 红旗信号

- 只有“管理员 / 普通用户”这类空泛角色
- 角色和部门混在一起，分不清岗位动作
- 说不清某一步具体是谁负责

### 追问方向

- 这个角色日常会看什么、填什么、确认什么
- 如果该角色不在，谁来顶替
- 是否有外部参与方，如甲方、分包、监理、政府部门
- 不同角色能看到的信息是否不同

### 本块至少要沉淀到 BRD 的内容

- 干系人地图
- 用户角色说明
- 各角色核心职责

---

## 维度 3：核心场景与业务流程

### 推荐推进方式：先总后分

先让产品或业务把这个域里所有核心场景 / 核心流程列出来，再逐条深挖。

推荐顺序：

1. 先列流程清单，通常 3-10 条
2. 给每条标注主流程 / 支线流程 / 异常流程，以及大致优先级
3. 先深挖主路径和 P0 流程
4. 再回到清单，补支线、例外和边界流程

如果一上来就只抓一条流程猛挖，很容易漏掉全域范围、上下游关系和优先级。

### 这一维要搞清楚什么

- 这个业务域的主流程有哪些
- 每条流程从哪里开始，到哪里结束
- 中间每一步谁做、做什么、用什么信息、产出什么结果

### 什么程度才算问透

- 已经形成该业务域的流程总览，而不是只聊了零散一条
- 能区分哪些是主路径、哪些是支线、哪些是异常处理
- 主流程闭环清楚
- 每步动作、责任人、输入输出清楚
- 审批、分支、并行、退回、撤销、超时等异常路径被覆盖

### 红旗信号

- 一上来就跳进某一条流程，但说不清整个域还有哪些流程
- 流程清单只是页面或按钮清单，不是业务过程
- 只会说“提交 -> 审批 -> 完成”
- 说不清驳回后怎么办
- 说不清异常和边界情况
- 只讲 happy path，不讲实际例外

### 追问方向

- 先让对方列出所有核心场景 / 流程，并按重要性排序
- 每条流程的起点、终点和完成标志是什么
- 触发条件是什么
- 第一步和最后一步分别是谁发起、谁结束
- 每一步要填什么信息，基于什么判断
- 审批标准是什么
- 驳回、撤销、重提、超时怎么处理
- 是否存在并行协作
- 流程是否会触发其他域动作

### 本块至少要沉淀到 BRD 的内容

- 流程总览和优先级
- 至少一条高优先流程的正常路径
- 关键审批节点
- 关键异常处理

### 图示建议

当文字或表格已经不足以表达复杂关系时，优先考虑补图：

- `flowchart`：表达流程步骤、分支、审批、驳回、回退
- `sequenceDiagram`：表达多角色协作顺序和信息传递
- `stateDiagram-v2`：表达本流程涉及的对象状态变化

图示是辅助理解和讨论的工具，不是独立的交付物。

---

## 维度 4：业务规则与约束

### 这一维要搞清楚什么

- 流程背后的业务判断规则是什么
- 什么条件会触发额外审批、限制或例外
- 哪些状态可以互转，哪些绝对不行

### 什么程度才算问透

- 关键规则可以写成判断逻辑
- 关键阈值、时限、公式、例外清楚
- 状态流转边界清楚

### 红旗信号

- 规则只停留在“看情况”“一般来说”
- 数值阈值不明确
- 例外情况没人负责
- 状态流转没有边界

### 追问方向

- 金额、数量、时间是否有阈值
- 审批条件是固定还是按场景变化
- 是否有正式制度、规范、审计要求
- 编码、命名、编号是否有规则
- 规则有没有例外

### 本块至少要沉淀到 BRD 的内容

- 关键计算规则
- 关键阈值与权限边界
- 核心对象的状态流转
- 编码或命名规则

---

## 维度 5：业务对象与信息

### 推荐推进方式：先列对象清单，再展开重点对象

先让产品或业务把这个域里的核心业务对象、关键单据、主要报表列出来，再逐个挑重点展开。

推荐顺序：

1. 先列对象总览，区分主对象 / 从属对象 / 参考对象
2. 标出哪些对象是当前域真正的核心对象
3. 先展开核心对象，再补关键单据、报表和历史数据
4. 最后再讨论迁移和数据来源

如果一上来就直接抠某个表单字段，很容易漏掉对象边界、对象关系和数据来源。

### 这一维要搞清楚什么

- 这个域里有哪些核心业务对象
- 它们之间是什么关系
- 这些对象目前以什么介质存在

### 什么程度才算问透

- 已经形成对象总览，而不是只聊了一个单据或一个表
- 核心对象列得出来
- 对象关系说得清
- 关键属性和核心单据清楚
- 数据来源、迁移和报表需求有基本认知

### 红旗信号

- 一上来就陷入某个对象的字段细节，却说不清这个域还有哪些对象
- 只谈流程，不谈数据对象
- 说不清哪些字段真正关键
- 说不清对象之间是一对多还是多对多
- 忽略历史数据和报表

### 追问方向

- 先让对方列出所有核心对象、关键单据和主要报表
- 哪些对象是主对象，哪些是附属对象，哪些只是参考资料
- 核心对象有哪些
- 每个对象最重要的属性是什么
- 这些数据现在放在哪里
- 有没有既有单据、表格、模板
- 数据量大概多少
- 哪些角色要看报表，按什么维度看

### 本块至少要沉淀到 BRD 的内容

- 对象总览和优先级
- 核心业务对象清单
- 对象关系
- 核心单据
- 报表和数据迁移概况

### 图示建议

当对象关系已经超出表格可读性时，优先考虑补图：

- `erDiagram`：表达对象之间的主从、引用、多对多关系
- `flowchart`：表达单据流转和信息流向

图示是帮助团队统一对象边界的辅助工具，不替代表格和文字定义。

## 维度 6：跨域协作与外部依赖

### 推荐推进方式：先画依赖地图，再展开关键链路

先让产品或业务把这个域和其他业务域、外部系统、外部单位之间的输入输出关系列出来，再挑关键依赖逐条深挖。

推荐顺序：

1. 先列依赖总览，区分上游域、下游域、外部系统、外部协作方
2. 标出哪些依赖是主链路、哪些是可选链路、哪些是风险点
3. 先展开主链路，再补异常和降级方式
4. 最后再讨论接口、文件、手工协作等交互方式

如果一上来只聊一个接口或一张导入表，很容易看不清整个域的上下游位置。

### 这一维要搞清楚什么

- 这个域和其他业务域如何相互依赖
- 是否存在外部系统、外部单位、第三方服务依赖

### 什么程度才算问透

- 已经形成依赖总览，而不是只知道零散一个接口
- 能说清本域输入来自哪里、输出给谁
- 能说清依赖断掉时谁会受影响
- 能说清与外部协作方的信息交换方式

### 红旗信号

- 只盯着某个系统对接点，却说不清整体上下游
- 把这个域当成完全孤立
- 不知道哪些数据来自别的域
- 不知道本域产出会卡住谁
- 忽略对外传递要求

### 追问方向

- 先让对方列出所有上游 / 下游 / 外部依赖
- 哪些依赖是主链路，哪些是边缘依赖
- 上游域给本域什么数据
- 本域要输出什么给下游域
- 是否需要对接 OA、财务、邮件、短信、电子签等
- 外部协作是手工、文件还是接口
- 对方不可用时业务如何继续

### 本块至少要沉淀到 BRD 的内容

- 依赖总览和优先级
- 跨域数据依赖
- 外部系统和外部协作方
- 关键依赖风险

### 图示建议

当跨域链路多、角色多、交互顺序重要时，优先考虑补图：

- `flowchart`：表达域与域之间的主链路
- `sequenceDiagram`：表达跨角色 / 跨系统的信息交换顺序

图示用于帮助大家看清上下游，不替代依赖说明表。

## 维度 7：范围与优先级

### 这一维要搞清楚什么

- 这个域里哪些内容必须先做，哪些可以后做
- 有没有硬时间点、关键里程碑、已知风险

### 什么程度才算问透

- 能区分 P0 / P1 / P2
- 能说清优先级背后的原因
- 能说清关键时间约束和主要风险

### 红旗信号

- 所有内容都说“重要”
- 没有明确先后顺序
- 不知道项目时间节点
- 风险只说“沟通风险”“技术风险”这种空话

### 追问方向

- 如果只能先做 3 件事，哪 3 件
- 哪些东西不做会影响上线或价值验证
- 是否有院内、客户、合同、审计等硬性节点
- 你最担心什么

### 本块至少要沉淀到 BRD 的内容

- 优先级分层
- 时间约束
- 已知风险

---

## 补充原则：不要停留在表面诉求

如果用户给的是功能口号，而不是业务问题，要继续追问。

常见表面诉求示例：

- 要 Excel 批量导入
- 要审批流
- 要大屏
- 要一键生成
- 要消息提醒

处理方式：

1. 先问清现在为什么会依赖这种方式
2. 再问真实痛点是什么
3. 再问这是不是日常主路径，还是补录 / 迁移 / 应急手段
4. 最后才把真实问题沉淀到 BRD
